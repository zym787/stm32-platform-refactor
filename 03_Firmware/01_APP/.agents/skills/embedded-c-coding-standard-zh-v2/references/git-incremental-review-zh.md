# Git 增量审查说明

## 为什么 V2 默认优先增量审查

对整个工程做全量扫描虽然覆盖面大，但在日常开发中通常存在三个问题：

- 资源开销大，耗时和上下文占用明显增加。
- 历史遗留问题会淹没本次改动引入的新问题。
- 自动生成代码、第三方代码和自研代码容易混在一起，降低审查信噪比。

因此，V2 默认把 Git 变更范围作为审查入口。

## 默认决策

### 场景 1：日常改动或提交前自检

默认选择：

- 审查 Git 变更涉及的自研 `.c/.h` 文件
- 补读当前函数、直接相关头文件和必要的共享定义

不要做：

- 全工程扫描
- 只看 diff 行不看上下文

### 场景 2：大规模重构或基线治理

默认选择：

- 按模块、目录或全工程做扩展审查

典型触发：

- 统一命名
- 批量格式整改
- 建立 CI 门禁
- 历史问题清理

### 场景 3：没有 Git 信息

默认选择：

- 退化为用户指定文件范围
- 若用户未指定，则只审当前任务直接涉及的文件

需要明确说明：

- 当前不具备 Git 基线
- 已改用局部范围审查

## 推荐 Git 范围选择

### 工作区未提交变更

```bash
git status --short
git diff --name-only
```

### 暂存区变更

```bash
git diff --cached --name-only
```

### 当前分支相对基线分支

```bash
git diff --name-only origin/main...HEAD
```

## 默认纳入与排除范围

### 默认纳入

- 项目自研模块（按项目实际目录结构识别，如应用层、服务层、平台层、BSP、工具库等自研目录）

### 默认排除

- `Drivers/`
- `03_Os/`
- 外部中间件
- `Core/` 下大多数生成代码
- 构建产物、日志和临时文件

## 审查深度边界

增量审查不等于浅层审查。默认做法是：

1. 用 Git 找到本次改动文件。
2. 在每个变更点回到所在函数或局部逻辑块。
3. 必要时补读：
- 对应头文件
- 直接相关宏、枚举、结构体
- 资源初始化与释放路径
- 直接受影响的调用点或接口定义
4. 聚焦输出本次变更引入的问题。

不建议退化为“只要不在 diff 行里就不看”，否则很容易漏掉：

- 返回值检查缺失
- 参数校验缺失
- 资源泄漏
- 并发访问风险
- 指针与长度类型问题
